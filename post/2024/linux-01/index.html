Linux

体系结构

  • 内核（kernal）

    • 管理硬件资源
      • CPU -> 进程调度模块
      • 内存 -> 内存管理模块
      • 外部设备 -> 文件管理，网络通信，设备驱动
    • 对上层应用程序提供接口->系统调用

  • 系统调用（system calls）

    • 内核给上层应用程序提供的接口

  • 库函数（library functions）

    • 对系统调用的包装

      给用户的接口 包装的系统调用
      malloc sbrk
      printf write

      不同系统相同功能的系统调用是不同的，封装包装好系统调用有以下优点

      • 方便使用
      • 可移植

Makefile原理

  • Makefile

    • makefile 是一个脚本文件，使用make工具可以解释执行makefile脚本文件

    • makefile 的优点

      • 自动编译
      • 增量编译（只编译新增和修改过的.c文件）
      a.c  -> a.o
b.c  -> b.o   
c.c  -> b.o   -> 链接生成 executable
···            
x.c  -> x.o

  • 书写方法

    makefile对语法要求非常严格

    algs.h

    
#ifndef __WD_ALGS
#DEFINE __WD_ALGS

int add(int a, int b);
int mul(int a, int b);
int dev(int a, int b);

    add.c

    #include "algs.h"

int add(int a, int b){
    return a + b;
}

    mul.c

    #include "algs.h"

int mul(int a, int b){
    return a * b;
}

    div.c

    #include "algs.h"

int div(int a, int b){
    return a / b;
}

    main.c

    #include "algs.h"

int main(int argc, char* argv){
    int a = 1, b = 3;
    return mul(3,4);
}

    Makefile

    main: main.o  add.o mul.o div.o ##目标文件和依赖文件  
	gcc main.0 add.o sub.o mul.o div.o -o main  ##shell命令
## 目标和命令共同构成了规则
main.o: main.c algs.h
	gcc -c main.c -Wall -g
add.o: add.c algs.h
	gcc -c add.c -Wall-g
sub.o: mul.c algs.h
	gcc -c mul.c -wall-g
div.o: div.c algs.h
	gcc -c div.c -Wall-g
	....
.PHONY: clean rebuild
clean:
	rm- rf *.o main
rebuild: clean main

  • 原理

    • 依赖关系: 一般使用有向无环图来表示（Directed Acyclic Graph, DAG）, make在执行脚本的时候，在内部会生成DAG，执行过程会用到DFS（深度优先遍历）

                 main 
        /   |   \              
   main.o add.o sub.o  ···
    / \    | |  ···  ··· 
main.c \   |add.c 
       algs.h

      通过拓扑排序可以保证有向无环图的依赖关系, 常常会用到DFS算法

    • 伪目标：规则中的shell命令不一定要生成目标文件

      .PHONY: clean    ##PHONY伪目标
clean: 
	rm-rf *.o main

      如上规则，没有任何目标文件

      $ make clean

      这样的伪目标可以反复执行

    • 什么情况下会执行命令？

      • 目标不存在
      • 依赖比目标更新

  • 效果

    $ make

    会有自动编译和增量编译的效果

库文件

什么是库文件：库文件是目标文件(*.o)的集合

分类：

  • 静态库：.lib .a 对函数的链接在连接阶段完成

    优点：

    • 移植方便

    缺点

    • 浪费空间，每一个进程中都有静态库的一个副本

      [进程 .lib]    [进程  .lib]

    • 对程序的更新，部署，发布不友好

  • 动态库：.ddl .so (dynamic link library) 在运行时完成对函数的链接

    优点：

    • 可以共享

    • 对程序更新，部署，发布友好

      [进程]   [进程]
     \   /
     [.so]

    缺点

    • 不方便移植

打包生成静态库的流程：

$ ls
args.h add.c  sub.c mul.c div.c  
$ make
或者
$ gcc -c add.c sub.c mul.c div.c
$ ar crsv libalgs.a add.o sub.o mul.o div.o   ##打包生成静态库
$ sudo mv libalgs.a /usr/lib
$ mv * tmp/
$ vim main.c

在main.c中使用

#incldue "algs.h"
